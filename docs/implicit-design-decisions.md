# 暗黙の設計上の決定事項 (Implicit Design Decisions)

本書は、仕様書や基本設計に明示的に記載されていなかったものの、実装・運用フェーズを通じて決定・確定し、アプリケーションの動作の中核となっている「暗黙の設計上の決定事項」を明文化したものです。

## 1. ランチャーにおける新規サイトの検知と配置順序 (Discovery Order)
リスキャン（再監査）によって新しく見つかった GitHub Pages サイトの配置アルゴリズムについて：
- **仕様**: 新規に発見されたサイト（まだ `launcherLayout` レコードにIDが存在しないサイト）は、ランチャー配列の「最後（Append）」ではなく「**最初（Prepend）**」に追加されます。
- **目的**: ユーザーが新たにデプロイしたり Pages を有効にしたリポジトリを、目につきやすい先頭に配置するためです。

## 2. Z-Indexと空間スタッキングの揮発性 (Ephemeral Z-Index Strategy)
ランチャーグリッド内の各タイルの重ね合わせ（`zIndex`）の管理について：
- **仕様**: データベース（Firestore）には Z-Index の値は一切保存されません。Z-Index は完全にクライアントのブラウザメモリ上で、要素の現在状態（ドラッグ中、プレス中、静止）と1次元の配列内インデックスに基づいて動的に計算されます。
  - 例: `isDragged ? 40000 + (1000 - baseIndex) : isPressed ? 50000 + (1000 - baseIndex) : 20000 + (1000 - baseIndex)`
  - 方向依存の Z-Index 補正（上方にドラッグした要素を前面に、下方へドラッグした要素を背面に）は純粋に UI 上の処理です。
- **目的**: 複雑な3D状態の永続化を避け、状態の競合や保存失敗による表示バグをゼロにするためです。

## 3. レスポンシブ性を担保する1次元レイアウトと設定の永続化 (1-Dimensional Layout & Settings Persistence)
タイルの配置状態および表示設定の保存データ構造について：
- **仕様**: Firestore (`settings/launcherLayout`) にはタイルの絶対座標（X, Yピクセル）は**保存されません**。純粋な1次元の文字列配列（例: `["owner/repo-a", "owner/repo-b", ...]`）としてIDの順序のみが保存されます。また、アニメーション速度 (`animationSpeed`) や表示アイコン数 (`visibleIconsRange`) といったユーザー設定も併せて保存されます。
- **目的**: ユーザーがモバイル（縦長）で開いた場合とPC（横長）で開いた場合とで、画面幅に応じて適切に「リフロー（折り返し配置）」が行われるようにしつつ、表示上の好みをデバイス・セッション間で維持するためです。

## 4. ファビコンとサイトメタデータ取得のプライバシー (Site Metadata Privacy & Best-Effort Fetch)
ランチャータイルに表示するアイコン（ファビコン等）の取得経路について：
- **仕様**: 外部のファビコンプロキシサービス（例: Google S2 Favicon 等）は一切使用しません。サイトメタデータの取得は、監査処理中に**Auditorバックエンド**によって実行されます。
- **実装詳細**:
  - 取得は完全に**認証情報フリー・PATフリー**であり、対象の GitHub Pages サイトに対して標準的な GET リクエストのみを送信します。
  - メタデータの収集はベストエフォートかつ非ブロッキングであり、失敗した場合は `/favicon.ico` へのフォールバックや null メタデータを返し、リポジトリの分類自体を失敗させることはありません。
- **目的**: 社内・プライベートなネットワークの URL やリポジトリ名が、外部の公開プロキシサーバーやファビコン集約サービスに漏洩（リーク）することを防ぐためです。

## 5. URL スキーマの厳格な排除 (Strict Protocol Allowlist)
ランチャーの遷移先等として生成・許可される URL の制限について：
- **仕様**: `ftp:`, `gopher:`, `mailto:` のようなカスタム・レガシースキーマは意図的に弾かれ、`null` ハンドリングされます。許可されるのは厳格に `http:` または `https:` のみです。
- **目的**: XSS や予期せぬローカルアプリケーションの起動（例: メールクライアントやftpクライアントの暴発）を防ぐためです。

## 6. SSL / HTTPS の状態表現の細分化 (Granular Error States for SSL)
Dashboard の表示における HTTPS 状態の表現について：
- **仕様**: 単純な「エラー」として一括りにせず、証明書が発行されているが強制（Enforce）されていない状態は「Approved but Not Enforced」として扱い、完全な証明書設定エラーや未設定状態である「SSL Configuration Issue」と視覚的・論理的に区別しています。
- **目的**: Github Pages 上の設定忘れ（Enforce のチェックボックスの押し忘れ）と、DNS レコードの CNAME エラー等による証明書発行自体の失敗をユーザーが直感的に切り分けられるようにするためです。

## 7. PWA 対応ステータスの永続化 (PWA Metadata Persistence)
ランチャータイルおよび監査結果における PWA メタデータの扱いについて：
- **仕様**: PWA の対応状況（`isPwa`, `pwaIconUrl`, `pwaName`, `pwaDisplayMode`）は、監査処理中にバックエンドで収集され、監査結果の一部として Firestore の監査キャッシュに保存されます。
- **実装詳細**: 
  - 永続化されたデータは、ランチャーの描画や詳細表示に使用されます。
  - PWA メタデータは監査時点のスナップショットであるため、対象サイトの更新後に監査を再実行しない限り、古い状態が表示される可能性があります。
- **目的**: 描画のたびにすべてのサイトに対して重複したメタデータ fetch を行う負荷を避け、保存された監査結果から一貫した情報を即座に表示するためです。

## 8. 自己完結型 UI アセット・ポリシー (Self-Contained UI Asset Policy)
アプリケーションの装飾やクロームに使用されるアセット（背景テクスチャ、アイコン等）の管理について：
- **仕様**: アプリケーション自体の物理的な UI デザインに使用される装飾的アセット（例: グリッド背景、UI パーツの画像）は、外部のサードパーティ製アセット URL に依存せず、ローカルのアセットまたは pure CSS による実装に限定されます。
- **区別**:
  - **監査対象メタデータ**: 監査によって発見された各サイトのアイコン（ファビコン、PWAアイコン）は、動的に発見される外部 URL です。
  - **アプリ・クローム**: アプリケーション自体の UI を構成するデザイン要素は、外部依存を排除します。
- **目的**: ネットワークの制約下での安定稼働を保証し、サードパーティ製アセットのリンク切れや、デザインパーツを通じた予期せぬトラッキング・外部リークを完全に防止するためです。

## 9. コンパクトなランチャー・メタデータ吹き出し (Compact Launcher Metadata Bubble)
ランチャータイルを長押しした際に表示される詳細情報（吹き出し）の UX 設計について：
- **仕様**: 吹き出しは情報を最小限に絞ったコンパクトな設計を採用しています。
  - **情報の取捨選択**: HTTPS の有無、Workflow 連携、PWA 対応といったポジティブなステータスはバッジとして表示されますが、「PWA非対応」といったネガティブな状態は視覚的ノイズを減らすために省略されます。
  - **揮発性**: 吹き出しの表示状態は永続化されず、ポインターイベント（長押し）によってのみ発生する揮発的な UI 要素です。
  - **クリーンアップ**: ポインターを離した（pointerup）際、およびドラッグが開始された際には、即座にタイマーをクリアして吹き出しを閉じる制御を行っています。
- **目的**: 密なグリッドレイアウトを維持しつつ、ユーザーが特定のサイトの技術的なステータスを素早く（クイックルック）確認できるようにするためです。

## 10. カスタムドメイン未設定時のリポジトリ名表示 (Repository-Name Default Pages Badge)
デフォルトの GitHub Pages プロジェクトサイトにおける円形バッジの表示について：
- **仕様**: カスタムドメインが未設定の Pages URL（例: `https://<owner>.github.io/<repositoryname>/` や `https://<owner>.github.io/`）に対して、円形に回転するテキストとして「**リポジトリ名（緑色）**」を使用します。
- **制約**: デプロイ元のソースブランチ名（`sourceBranch`）はUIのバッジ文字列として表示しません。カスタムドメイン設定済みのサイトは引き続きドメイン名を元にしたテキストを表示します。
- **目的**: これは純粋な視覚的プレゼンテーション（UI表現）のみの変更であり、監査のロジック、書き出しスキーマ、APIの呼び出し回数、または Firestore の保存データには一切影響を与えず、安全にユーザーの視認性を高めるためです。

## 11. ダイレクト-DOM 物理レンダリングと IntersectionObserver (Direct-DOM Physics Rendering & IntersectionObserver)
ランチャーにおけるドラッグアニメーションおよび円形テキストアニメーションのパフォーマンス最適化について：
- **仕様 (Direct-DOM Rendering)**: ランチャーの物理演算（ドラッグ操作による高頻度な位置更新）は、対象要素の絶対位置（`el.style.transform`）を DOM に直接適用（Mutate）することで実現しています（`requestAnimationFrame` ループを利用）。React の状態（State）は論理的な順序や永続化の同期のみを担当し、毎フレームの再描画には関与しません。
  - 各ランチャーカードには識別子として静的なDOM ID（`launcher-node-${site.id}`）が必要とされます。
  - ドラッグ終了時または構造的変化時にのみ、最終的な順序が React に同期されます。
- **仕様 (IntersectionObserver)**: 円形テキスト（Spinning Badge）のアニメーションは、画面外（オフスクリーン）の場合には IntersectionObserver を用いて一時停止される実装となっています。現在の仕様では厳密に `rootMargin: '0px'`, `threshold: 0` を使用してビューポートの可視境界を判定し、早期のプリロードマージンを利用しません。
- **目的**: 状態更新による無駄な React 再レンダリングとレイアウト・スラッシングを完全に排除し、滑らかで高いフレームレートでの物理シミュレーションを可能にすると同時に、描画不可視領域のCPU消費を削減する純粋なパフォーマンス最適化です。これらの情報は UI ステートとして揮発性であり、Firestoreの設定（`settings/launcherLayout`）には書き込まれません。ビューポート外に出たタイミングで厳密に一時停止されます。

## 12. サーバーサイド画像キャッシュと永続化ポリシー (Server-side Image Caching & Persistence Policy)
ランチャータイルに表示するカスタムアイコン（ファビコン、PWAアイコン等）のキャッシュ機構について：
- **仕様**: 
  - **サーバーサイドでの画像解決理由**: ブラウザ側での直接の画像取得はCORS（Cross-Origin Resource Sharing）制約によってブロックされるため、バックエンド (`/api/icon/resolve`) が代理で画像バイナリをフェッチし、Base64文字列に解決してレスポンスします。
  - **Firestore キャッシュパス**: 
    - ログインユーザー: `githubPagesAuditorV2/{environment}/users/{uid}/launcherIconCache/{cacheId}`
    - 匿名（ゲスト）ユーザー: `githubPagesAuditorV2/{environment}/anonymousSessions/{uid}/launcherIconCache/{cacheId}`
  - **キャッシュドキュメント構造**:
    - `siteId`: 対象の監査サイトID
    - `sourceIconUrl`: 取得元のアイコンURL
    - `sourceKind`: キャッシュ元のアセット種別 (`pwa_icon` または `favicon`)
    - `contentType`: `image/png` / `image/jpeg` などのMIMEタイプ
    - `dataBase64`: Base64エンコードされた画像のペイロード
    - `byteLength`: 画像のバイトサイズ
    - `sha256`: ペイロードのSHA256ハッシュ値
    - `fetchedAt`: 取得日時（ISO文字列）
    - `expiresAt`: キャッシュ有効期限（ISO文字列、取得時から30日後）
  - **30日更新ポリシー (Stale-While-Refresh)**: キャッシュの有効期限は30日間です。有効期限 (`expiresAt`) を過ぎたキャッシュへのアクセスが発生した際、または有効期限内であっても30日が経過している場合、stale-while-refreshポリシーに基づき、既存をそのまま表示しつつ非同期（バックグラウンド）で再取得を行い、最新情報に更新します。SVG画像のバイナリキャッシュは、サイズコントロールやスクリプトインジェクション耐性の観点から「v1.7.35」では意図的に対象外（Raster画像のみ）としています。
  - **限定的なリゾルバー**: `POST /api/icon/resolve` は汎用的な外部Webプロキシではなく、ランチャーアイコンの取得に特化した安全なリゾルバーです。
  - **フォールバック設計（ベストエフォート）**: キャッシュの読み込みエラーやバックエンドのリゾルバー接続失敗が発生した場合でも、ランチャーの描画がクラッシュしたり、ユーザーへのエラー画面を表示することは一切ありません。既存の Launcher 表示は完全に機能し続け、ベストエフォートで元の外部URL（SSL経由等）またはローカルで生成されたイニシャルバッジへと段階的にフォールバックします。
- **目的**: サイトの描画負荷および外部への余計なアクセスを削減しつつ、CORS制約に干渉されない安定した表示と堅牢なセキュリティ境界（SSRF防止・容量制限・SVGの安全な排除等）を両立させるためです。
